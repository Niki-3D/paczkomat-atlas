# PostGIS Conventions

## SRIDs
- Storage: 4326 (geography for points).
- PL distance ops: 2180 (PUWG 1992 / Poland CS92).
- EU pan-region: 3035 (LAEA Europe).
- Web Mercator (3857): tile generation only.

## Tables
- `lockers (id, name, country, geom geography(Point,4326), gmina_id, nuts2_id, raw jsonb, ...)`
- `gminy (teryt PK, name, geom geometry(MultiPolygon,2180))`
- `nuts2 (code PK, name, country, geom geometry(MultiPolygon,4326))`
- `population_gmina (teryt FK, year, value bigint)`
- `population_nuts2 (code FK, year, value bigint)`
- `mv_density_gmina` — materialized: lockers per 10k inhabitants per gmina
- `mv_density_nuts2` — same for EU

## Indexing

- **Geometry/geography columns**: rely on GeoAlchemy2's `spatial_index=True` (default) which emits a DDL after-create hook to install the GIST index. DO NOT also add `op.create_index(... postgresql_using='gist')` in the migration for the same column — Alembic autogenerate will try to emit it, but you must DELETE that line before applying the migration. The autogenerated index will collide with GeoAlchemy2's and fail with `relation already exists`.
- After autogenerate, ALWAYS scan the new migration for `op.create_index` calls with `postgresql_using='gist'` on geom columns. Remove them. Leave a comment documenting why.
- BRIN on `created_at` / `updated_at` columns when the table grows beyond ~1M rows.
- B-tree on every FK column referenced in joins (`gmina_teryt`, `nuts2_id`).
- Composite indexes for known filter combinations — e.g. `(country, is_locker)`, `(country, status)`.

## Operators
- Radius: `ST_DWithin(geom, ST_MakePoint(lon,lat)::geography, meters)`.
- Point-in-polygon: `ST_Within(ST_Transform(p.geom::geometry, 2180), g.geom)`.

## Materialized views
- All density/aggregation queries hit MVs.
- Refresh: `REFRESH MATERIALIZED VIEW CONCURRENTLY mv_*` (needs UNIQUE index).
- Cron daily after ingest.
- MVs filter on `status IN ('Operating', 'Overloaded')` — both are valid operational states per data-quality-rules. NEVER include `Created`, `Disabled`, or `NonOperating` in dashboard-facing MVs.
- pg_cron jobs that REFRESH MVs are scheduled in Europe/Warsaw timezone (set globally on the cluster). Schedule with at least 5-minute gaps to avoid contention.

## Alembic + pgbouncer

- Alembic runs against the **direct DB connection** (port 5432), NOT through pgbouncer (port 6432). Migrations need transactional DDL and advisory locks, which pgbouncer transaction-mode pooling breaks.
- Application runtime uses pgbouncer (port 6432) with `prepared_statement_cache_size=0` — auto-detected in `db.py` when the URL contains `pgbouncer` or `:6432`.
- The `.env` `DATABASE_URL` points at direct connection (:5432). Production deploys override at runtime via `docker-compose.prod.yml`.

## include_object env.py filter

Alembic's autogenerate sees PostGIS and TimescaleDB internal objects (`spatial_ref_sys`, `topology.layer`, `_timescaledb_internal.*`). The `include_object` callback in `alembic/env.py` filters them out. When adding a new system schema, extend the excluded set in `env.py`.
